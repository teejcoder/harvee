# Changelog

## [Unreleased]

### Added

- Initial repo skeleton: SvelteKit 2 + Svelte 5 (runes) + TypeScript strict + Tailwind v4 on Node 22, with Prettier, ESLint, and Vitest configured.
- Vitest test workspaces: `unit` (Node) and `component` (jsdom, `@testing-library/svelte`). Test root at `tests/{unit,component,e2e}/`.
- Playwright end-to-end test harness (`playwright.config.ts`), booting against `pnpm preview`.
- `@sveltejs/adapter-vercel` as the production adapter (deploy target: Vercel).
- VitePress documentation site at `docs/`, with mermaid support (`vitepress-plugin-mermaid`). Nav sections: Architecture, Guides, Decisions, Changelog. Stub pages under each section point at `.memory/` for the current source of truth. Scripts: `docs:dev`, `docs:build`, `docs:preview` (port 5174).
- ULID module at `src/lib/ids.ts` — thin `monotonicFactory()` wrapper around the `ulid` package; single source of ULIDs for every entity ID and every correlation ID. Covered by two Vitest tests (regex format + 10k ascending/unique).
- Pure-utility logging exemption in `.memory/conventions.md` §6 and `CLAUDE.md` §4.2.1: modules with no I/O, no side effects, and no branch decisions are exempt from the "every function logs" rule. Currently covers `src/lib/{ids,time,money}.ts`.
- Time module at `src/lib/time.ts` — `nowUtcIso`, `localDateOf`, `localDayBounds`, `localWeekBounds` (Monday-anchored), `localMonthBounds`. All timezone math lives here; day/week/month boundaries return half-open UTC ISO intervals. Optional `tz` argument on every function (defaults to system local). Covered by 11 Vitest cases spanning DST forward (23h day), DST back (25h day), and non-DST (24h day) zones.
- Structured logger at `src/lib/log.ts` — `log.debug/info/warn/error({...})` and `logTransition({...})`. camelCase JSON lines, appended to `logs/transitions.jsonl`. Exports `REJECTION_REASONS` (canonical list) and a discriminated `TransitionEntry` union so accepted/rejected mismatches are compile-time errors. `LOG_PATH` env var overrides the default file location for tests. Covered by 5 Vitest cases.
- Correlation-ID hook at `src/hooks.server.ts` — every non-GET / non-HEAD request mints a ULID, stores it on `event.locals.correlationId`, and attaches it to the response as `X-Correlation-Id`. Read paths (GET, HEAD) get no correlation ID per `.memory/conventions.md` §5. `App.Locals.correlationId` typed in `src/app.d.ts`. Covered by 6 Vitest cases (POST/PUT/PATCH/DELETE mint + attach, GET/HEAD skip, uniqueness across calls).
- SQLite connection + migration runner at `src/lib/db/index.ts`. `openDb(path?, migrationsDir?)` opens a database and applies every `.sql` file in `db/migrations/` in ascending filename order inside a transaction, tracked in a `_migrations` table. Already-applied files log `db.migrate.skip` and are no-ops. `getDb()` returns the memoized production connection (`./data.sqlite`). Missing migrations directory logs a warn but does not fail. WAL + foreign-keys pragmas enabled at open. Warmed at server boot by `src/hooks.server.ts` so migrations run before the first request. Covered by 5 Vitest cases including a transaction-rollback assertion. Runtime dep: `better-sqlite3`.
- First migration `db/migrations/001_settings.sql` — creates the singleton `settings` table (`INTEGER PRIMARY KEY CHECK (id = 1)`) with sender identity, currency, default payment terms, and invoice locale per `.memory/domain-model.md` §4. Seeds one row with placeholder values so `/settings` has something to render on first boot. Covered by 4 Vitest cases (CHECK constraint, placeholder shape, second-row rejection, second-boot idempotency).
- Second migration `db/migrations/002_clients_projects_tasks.sql` — three ULID-keyed tables per `.memory/domain-model.md` §10. `clients (id, name, archived_at?, timestamps)`. `projects (id, client_id → clients.id, name, hourly_rate INTEGER minor units, archived_at?, timestamps)` with index on `client_id`. `tasks (id, project_id → projects.id, name, archived_at?, timestamps)` with index on `project_id`. FKs default to NO ACTION (behaves like RESTRICT), matching the "block delete if children" rule. Covered by 7 Vitest cases (per-table column shape, FK metadata, orphan-insert rejection, delete-with-children rejection, indexes, second-boot ordering).
- Third migration `db/migrations/003_invoices.sql` — invoices + invoice_line_items per `.memory/domain-model.md` §5/§6/§8/§10. Invoices carry state (draft/finalized/exported/voided), date range, nullable UNIQUE `invoice_number`, payment terms, snapshotted currency triple, and three integer-minor-unit money fields, with CHECK constraints enforcing state-dependent nullability (invoice_number/finalized_at/voided_at) plus `total = subtotal + discount_total` and `discount_total <= 0`. Line items carry kind (task|discount), nullable task FK, description, hours/rate/amount/sort_order, with a kind-specific field-shape CHECK and a partial UNIQUE INDEX enforcing at most one discount line per invoice. `ON DELETE CASCADE` on `invoice_id` so draft-delete cleans up line items. Covered by 18 Vitest cases.
- Migration ordering swap in `.memory/implementation-plan.md`: invoices now Step 1.4 (was 1.5) so `time_entries.invoice_id → invoices.id` FK works — SQLite with `foreign_keys = ON` rejects INSERTs into tables that FK-reference a not-yet-created table, even for NULL FK values.
- Fourth migration `db/migrations/004_time_entries.sql` — `time_entries` (task FK, notes, state ∈ 6 lifecycle states, nullable `invoice_id → invoices.id`, nullable `edit_form_snapshot` JSON, timestamps) with a CHECK enforcing `state = 'entry.locked'` iff `invoice_id IS NOT NULL`. `time_entry_segments` (entry FK, started_at, nullable stopped_at) with CHECK `stopped_at IS NULL OR stopped_at >= started_at` and a partial UNIQUE INDEX ensuring at most one open segment per entry. Covered by 15 Vitest cases.
- Test-isolation guard in `src/hooks.server.ts`: the module-scope `getDb()` warmup is skipped when `process.env.VITEST` is set, so importing the module from a Vitest run no longer touches `./data.sqlite`. Rule codified in `.memory/conventions.md` §11.
- Query modules at `src/lib/db/queries/{settings,clients,projects,tasks,entries,segments,invoices,lineItems}.ts` plus `_helpers.ts`. Each exports typed interfaces, one or more write functions (require `correlationId: string`; runtime-checked, logs and throws on undefined), and read functions (no cid). All results are camelCase. Prepared statements cached per DB connection via WeakMap. Invoices module exposes only `createDraftInvoice` — non-draft states enter via state-machine transitions in Phase 2. Settings uses `getSettings` + `updateSettings` (no create, singleton). Covered by 18 Vitest cases (per-entity insert-then-read camelCase mapping + a `test.each` over all 9 write functions asserting undefined-cid → throws + ERROR log line).
- `EntityType` in `src/lib/log.ts` gained `'settings'` — was missing, TypeScript surfaced it during Step 1.6.
- Doc realignment: query modules moved from `db/queries/` to `src/lib/db/queries/` in `.memory/tech-stack.md`, `.memory/conventions.md` §2/§7, and `.memory/implementation-plan.md` Step 1.6. Rationale: keep TS files inside SvelteKit's default tsconfig `include` scope.
- Client / Project / Task state machines at `src/lib/state/{client,project,task}.ts` per `.memory/state-transitions.md` §1. Each exposes `create*`, `archive*`, `unarchive*`, and `delete*`. Every operation runs preconditions, applies the DB write, and emits an accepted OR rejected transition-log line. `StateTransitionError` carries the typed `rejectionReason`. Enforced rejection codes: `parent_archived` (create under archived parent), `children_not_archived` (archive with active children), `task_has_running_timer` (archive task with a running time_entries row), `referenced_by_invoice` (delete when any non-draft invoice references the entity). Covered by 15 Vitest cases including a coverage self-check asserting every §1 rejection code fires at least once.
- Pseudo-terminal states `client.deleted` / `project.deleted` / `task.deleted` added to `.memory/state-transitions.md` §1 — they only appear in transition-log lines (successful or attempted hard-delete), so delete attempts are visible in the log with a canonical `newState`.
- Time entry state machine at `src/lib/state/entry.ts` per `.memory/state-transitions.md` §2. Ten operations: `pickTask`, `startTimer`, `stopTimer`, `openEdit` (snapshots segments to `edit_form_snapshot`), `saveEdit`, `cancelEdit` (restores from snapshot), `resumeEntry` (opens a new segment on a stopped entry), `updateSegment`, `discardEntry`, plus system-triggered `lockEntry` and `unlockToDiscarded` (called by the invoice state machine in Step 2.3). All six §2 rejection codes enforced: `concurrent_timer_forbidden`, `entry_locked_by_invoice`, `invalid_time_range`, `segment_overlap`, `task_archived`, `cannot_edit_running_entry`. Segment mutations use `log.warn` (not `logTransition`) since segments have no state column. Covered by 16 Vitest cases including a Start → Stop → Resume → Stop → 2-segments scenario and a full coverage self-check across all six rejection codes.
- Invoice state machine at `src/lib/state/invoice.ts` per `.memory/state-transitions.md` §3 and `.memory/domain-model.md` §5/§6/§8. Seven operations: `generateDraftInvoice` (queries eligible entries, groups by task, snapshots project rate, creates draft + task lines in one transaction), `addDiscountLine`, `removeDiscountLine`, `finalizeInvoice` (assigns `YYYYMMDD-N`, cascades every eligible entry to `entry.locked` under one correlationId), `exportInvoice`, `voidInvoice` (cascades every locked entry to `entry.discarded` under one correlationId), `deleteDraft` (targets pseudo-terminal `invoice.deleted`). All six §3 rejection codes enforced: `no_billable_entries`, `invoice_locked`, `invoice_non_positive_total`, `invalid_discount_line`, `must_finalize_before_export`, `void_requires_finalized`. Covered by 16 Vitest cases including finalized→voided and exported→voided cascades.
- Pseudo-terminal state `invoice.deleted` added to `.memory/state-transitions.md` §3, matching the client/project/task convention from §1.
- Step 2.4 gate: end-to-end scripted scenario test at `tests/unit/lib/state/end-to-end.test.ts` — creates client → project → task, picks task → starts → stops → opens edit → saves → generates draft invoice → finalizes, and asserts the resulting transition-log sequence against an explicit expected array (previousState, newState, trigger, actor.type, accepted). Verifies the finalize cascade fires `entry.stopped → entry.locked` (system) INSIDE the transaction before the outer `invoice.draft → invoice.finalized` (user) line.
- Logger self-exemption in `.memory/conventions.md` §6: `src/lib/log.ts` has I/O but cannot log its own entry without infinite recursion.
